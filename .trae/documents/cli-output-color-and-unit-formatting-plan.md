# CLI 输出格式化与交易日自动延顺计划

## 概述

针对用户提出的两个需求：

1. **颜色与单位格式化**：涉及价格的地方标注颜色（涨红跌绿），成交量/成交额添加单位（万/亿）并保留3位小数
2. **交易日自动延顺**：输入的开始日期如果不是交易日，自动延顺到下一个交易日

对 `main.py`（展示层）和 `data_fetcher.py`（数据层）进行修改。

## 变更计划

### 变更 1：`data_fetcher.py` — 添加交易日自动调整逻辑

**新增方法 `_adjust_trading_day(self, date_str, mode="next")`**：
- 调用 `ak.tool_trade_date_hist_sina()` 获取全量A股交易日历（已排序）
- 将结果缓存在实例变量 `self._trade_calendar` 中（避免重复API调用）
- `mode="next"`（开始日期用）：如果不是交易日，查找下一个最近的交易日（向前找）
- `mode="prev"`（结束日期用）：如果不是交易日，查找上一个最近的交易日（向后找）
- 如果是交易日 → 直接返回原日期
- 如果 date_str 超出交易日历范围 → 直接返回原日期（akshare会自行处理）

**修改 `get_history_data()`**：
- 在调用 `ak.fund_etf_hist_em()` 之前：
  - 对 start_date 执行 `mode="next"` 调整（延顺到下一个交易日）
  - 对 end_date 执行 `mode="prev"` 调整（回退到上一个交易日）
- 记录日志：`"开始日期 %s 不是交易日，自动延顺至 %s"` / `"结束日期 %s 不是交易日，自动回退至 %s"`

## 决策说明

1. **交易日逻辑放在** **`data_fetcher.py`**：因为所有调用 `get_history_data()` 的地方（main.py 的多个菜单、analyzer.py 的分析方法）都能自动受益，无需在每个调用点重复实现
2. **交易日历缓存会话级**：在 ETFDataFetcher 实例内缓存交易日历，避免每次查询都调用 akshare API
3. **价格着色基准**：实时行情中 open/high/low/price 与 prev\_close 比较；涨跌幅/涨跌额直接根据正负值
4. **不引入第三方库**：颜色使用内置 ANSI 码，与现有风格一致
